# Design System: HEIL TECH

## 1. Overview

**Creative North Star: "The SCIF Terminal"**

This system is modeled on a Sensitive Compartmented Information Facility workstation: physically present, hardware-real, operating under perpetual constraint. Every surface is dark not because dark looks cool but because this terminal lives in a windowless room with no ambient light. Every type choice is monospaced or hyper-compressed black because the operators reading it move fast and tolerate nothing decorative. The crimson is not an aesthetic accent — it is an alert state, a blood pressure reading, a mission-critical flag.

The system rejects the aesthetics of hacker theater (Matrix rain, neon on black, glitch effects deployed for personality) and corporate IT reassurance (navy and gray, stock handshake photography, clip-art shields). What remains is aerospace/defense documentation meets terminal-native UI: classified memo typography, phosphor-glow data readouts, CRT grain baked into the substrate. The interface itself is the evidence of competence. No "trusted by" badge grid can do what a well-engineered HUD does.

New screens should be generated by asking: "Would this element appear in a SCIF?" If it's decorative, it shouldn't. If it's functional, it should look like it's been stress-tested.

**Key Characteristics:**
- Dark without relief: the void is deep, surfaces layer in three tight steps
- Crimson reserved for live state: active threats, CTAs that must act now, danger indicators
- Type hierarchy through scale extremes: display at 8.5rem, labels at 0.625rem — nothing comfortable in between
- All-caps monospace for every metadata field, timestamp, badge, and label
- Zero border-radius: no soft edges, no pill shapes — every corner is a right angle
- Phosphor glow on accent text: text-shadow renders data like it's emitting light
- CRT grain and scanlines baked into the global stylesheet as body pseudo-elements
- WebGL shaders (radar sweep, noise field, signal wave) as functional infrastructure, not decoration

## 2. Colors: The Operations Palette

A three-step void stack, one alert color, and a careful zinc gradient for text — nothing invented, nothing decorative.

### Primary
- **Alert Crimson** (`#b91c1c`): The system's single accent. Used for live-state indicators, primary CTAs, active danger flags, structural borders on the most important UI elements. Its rarity is proportional to its urgency. When every CTA is crimson, nothing is urgent.
- **Dimmed Crimson** (`#7f1d1d`): Hover/pressed state for crimson elements. Also used for phosphor bloom backgrounds on heavy glow effects.

### Neutral
- **Tactical Void** (`#09090b`): The base. Every background begins here. Not pure black; the faint blue-gray tint keeps it from reading as a printout.
- **Carbon Surface** (`#0f0f12`): First elevation above void. HUD panels, card containers, input backgrounds — anything that needs to appear one level above the substrate.
- **Hardened Shell** (`#161618`): Second elevation. Active elements, selected states, hover backgrounds on surface-level items.
- **Primary Text** (`#fafafa`): Headlines, CTA labels, any text carrying a command or designation.
- **Secondary Text** (`#a1a1aa`): Body copy, descriptions, supporting content. The reading register.
- **Muted Text** (`#52525b`): Timestamps, metadata, tags, dividers with text. The infrastructure register.
- **Dim Text** (`#3f3f46`): Separator text, decorative monoline, borders that should almost disappear.

### Named Rules
**The Blood Pressure Rule.** Crimson (`#b91c1c`) is used for three categories only: live-state indicators (active threat, system operational), primary CTAs (one per screen section, maximum), and structural emphasis (the one border that must not be missed). On any given screen, crimson covers less than 10% of the surface area. Its rarity is the point. A screen drowning in red has no emergency.

**The Tonal Depth Rule.** Depth is conveyed through the three-step void stack (`#09090b` → `#0f0f12` → `#161618`), never through shadows or blurs. A surface is elevated because its background is one step lighter, not because it casts a shadow. Shadows are prohibited.

## 3. Typography

**Display Font:** Geist (with system-ui, sans-serif fallback)
**Body Font:** Geist (same family, lower weight)
**Label/Mono Font:** Geist Mono (with monospace fallback)

**Character:** Geist's compressed proportions and tight spacing read as precision engineering, not approachability. Geist Mono carries classification labels and telemetry data — it implies the text came from a machine, not a copywriter. The pairing enforces the distinction between *what matters* (Sans, scaled to structural extremes) and *what it means* (Mono, stamped in small caps at the data layer).

### Hierarchy
- **Display** (900 weight, `clamp(3.5rem, 10vw, 8.5rem)`, line-height 0.9, tracking `-0.03em`): Section-defining headlines only. ELITE / CYBER / DEFENSE. Never used for subheadings. When multiple display lines stack, they compress into a solid typographic block.
- **Headline** (800 weight, `clamp(1.75rem, 4vw, 2.5rem)`, line-height 1.1, tracking `-0.02em`): Section titles: ACTIVE CAPABILITIES, TACTICAL CYBER EXPERTISE. One per section, uppercase.
- **Title** (700 weight, `~1rem–1.125rem`, line-height 1.2): Service card titles, panel headers. Bold sans, uppercase, no mono.
- **Body** (400 weight, `1rem`, line-height 1.6, `text-zinc-400` / `#a1a1aa`): Descriptions and prose. Max line length 65ch. Never in pure white; body text in secondary tone (`#a1a1aa`) so display headlines read louder by contrast.
- **Label** (400 weight, `0.625rem–0.75rem`, tracking `0.2em–0.25em`, Geist Mono, UPPERCASE): All metadata: timestamps, clearance badges, section codes (OPS-001), coordinates, status indicators, nav links. Never sentence-case. Never a display or body font.

### Named Rules
**The Two-Register Rule.** Every text element is either Sans (structural, commanding) or Mono (telemetric, systemic). Mixing them on the same surface without clear role distinction creates noise. The rule: if it could appear in a classified memo header or radar readout, it's Mono; if it's what the memo says, it's Sans.

**The No-Comfort Zone Rule.** The type scale has no comfortable mid-range. Display is enormous; labels are tiny. There is no `1.25rem` headline hovering between the two. This scale creates maximum visual distance between hierarchy levels, which is how operational data gets parsed at a glance.

## 4. Elevation

This system is flat by doctrine. No box shadows exist anywhere in the codebase. Depth is conveyed exclusively through the three-step background tonal stack: `void` → `surface` → `surface-up`.

A panel is elevated above the page not because it casts a shadow but because its background (`#0f0f12`) is lighter than the page beneath it (`#09090b`). The HUD overlay in the hero is elevated not by a drop shadow but by a thin `border border-white/5` that marks its boundary against the darkness. This makes depth physical and structural rather than ambient and decorative.

**The Flat-By-Default Rule.** Shadows are prohibited. Every instance of `box-shadow` is wrong. Depth exists because surfaces have different background luminosities, not because they appear to float. The only acceptable shadow is `text-shadow` on phosphor-glow utility classes (`.phosphor-red`, `.phosphor-white`, `.phosphor-green`), where it simulates emitted light, not surface elevation.

### Tonal Vocabulary
- **Ground** (`#09090b`): Page substrate. Never used as a component background above ground level.
- **Raised** (`#0f0f12`): One step above ground. Panels, HUD containers, card backgrounds.
- **Lifted** (`#161618`): Two steps above ground. Active/hover states on raised surfaces.

## 5. Components

### Buttons

The primary CTA is the only element on the page that can be physically pushed. It should feel like a physical toggle, not a web affordance.

- **Shape:** Zero radius (0px). Every corner is 90 degrees.
- **Primary:** Crimson fill (`#b91c1c`), white text (`#fafafa`), Geist Mono uppercase, `0.075em` tracking, `14px 28px` padding. On hover: dimmed crimson (`#7f1d1d`). On active: `scale(0.98)` tactile compression.
- **Ghost:** Transparent background, `border border-white/10`, secondary text (`#a1a1aa`). On hover: `border-white/25`, primary text (`#fafafa`). No fill, no color.
- **CTA Rule:** One primary button per section. Never two crimson CTAs competing in the same visual field.

### Tags / Chips

Classification labels and capability tags. Not interactive by default.

- **Style:** Transparent background, `border border-white/10`, Geist Mono uppercase at `0.625rem`, letter-spacing `0.1em`, text in muted tone (`#52525b`). Padding `2px 8px`.
- **No hover states** on non-interactive chips. Chips are data, not actions.

### Cards / Containers

HUD panels and service containers. Reads as physical enclosure, not digital card.

- **Corner style:** Zero radius (0px). Right angles only.
- **Background:** Carbon Surface (`#0f0f12`) or Hardened Shell (`#161618`) depending on depth position.
- **Shadow strategy:** None. Boundary defined by `border border-white/5` (barely visible) or by tonal contrast with surrounding surface.
- **Border:** `border-white/5` at rest. `border-crimson` only when the element requires maximum structural emphasis.
- **Internal padding:** `24px` baseline, `48px` for hero-scale panels.
- **Corner brackets:** Decorative `border-t-2 border-l-2 border-crimson` spans applied at the four corners of mission-critical panels (HUD frame treatment). These are structural markers, not decoration.

### Inputs / Fields

- **Style:** `border border-white/10`, Carbon Surface background (`#0f0f12`), zero radius. Geist Sans body text.
- **Placeholder:** Muted text tone (`#52525b`).
- **Focus:** `border-crimson/60` — the border warms toward alert crimson on active input.
- **Submit:** Primary button immediately adjacent. Never floating.

### Navigation

- **Style:** Fixed header, transparent at scroll=0, transitions to Tactical Void (`#09090b`) + `border-b border-white/5` after 40px scroll.
- **Logo:** Heil Tech shield mark + wordmark at `38px` height.
- **Links:** Geist Mono uppercase, `0.625rem`, `0.2em` tracking, secondary text. On hover: primary text + underline expands left-to-right via CSS `width: 0 → 100%` transition, crimson fill.
- **CTA:** Ghost button (see Buttons above) in the header. Never the crimson primary.
- **Mobile:** Slide-down drawer from `top: 72px`, Tactical Void background, same type treatment.

### Signature Component: WebGL HUD Panel

The radar sweep, noise field background, and signal wave dividers are first-class design elements, not Easter eggs.

- **RadarSweep:** Full-screen-equivalent canvas inside the hero HUD panel. Rotating phosphor sweep, concentric rings, 5 contact blips with exponential phosphor decay. Aspect ratio locked 1:1 in an `aspect-square` container.
- **NoiseField:** Absolute-positioned background canvas behind the hero. FBM noise + horizontal scanner band + vertical interference stripes. DPR capped at 1.5 for performance.
- **SignalWave:** `h-20` divider canvas between sections. Three overlapping sine waveforms (primary crimson, secondary dim, tertiary ghost) on Tactical Void background.
- **Rule:** WebGL canvases are always `pointer-events-none`. They never intercept clicks. They are atmospheric infrastructure.

### Phosphor Glow Utilities

Three utility classes for emitting-light text. Applied to data values only, never body copy.

- `.phosphor-red`: `text-shadow: 0 0 10px rgba(185,28,28,0.8), 0 0 3px rgba(185,28,28,0.5)`. Used on active threat counts, live status indicators, crimson accent data.
- `.phosphor-white`: `text-shadow: 0 0 8px rgba(255,255,255,0.7), 0 0 2px rgba(255,255,255,0.4)`. Used on key metrics (uptime, response time) in the HUD.
- `.phosphor-green`: `text-shadow: 0 0 8px rgba(52,211,153,0.9), 0 0 3px rgba(52,211,153,0.6)`. Used exclusively for the single SYSTEMS OPERATIONAL status indicator.

## 6. Do's and Don'ts

### Do:
- **Do** use Geist Mono uppercase at `0.625rem` / `0.75rem` with `0.2em` tracking for all metadata, labels, badges, codes, and navigation links.
- **Do** express depth through the tonal void stack (`#09090b` → `#0f0f12` → `#161618`). Background luminosity = elevation. No shadows.
- **Do** apply corner-bracket treatments (`border-t-2 border-l-2 border-crimson` on `w-5 h-5` spans) to mission-critical panels. This is the one decoration permitted because it reads as structural framing, not styling.
- **Do** use `scale(0.98)` on `:active` states for primary buttons. The physical compression is the feedback.
- **Do** restrict crimson to three use cases: live-state indicators, primary CTAs (one per section), and the single most important structural border per panel.
- **Do** animate only `transform` and `opacity`. Never `width`, `height`, `top`, `left`.
- **Do** use `min-h-[100dvh]` for the hero. Never `h-screen`.
- **Do** use `display: grid; gap: 1px` with contrasting parent/child backgrounds to create razor-thin dividers without border declarations.

### Don't:
- **Don't** use Matrix rain, neon green on black, glitch effects deployed for personality, or hooded-figure imagery. This is the cheap hacker aesthetic explicitly excluded in PRODUCT.md. It signals amateur, not elite.
- **Don't** use purple, teal, or gradient hero backgrounds. Generic SaaS aesthetics destroy the military precision the brand requires.
- **Don't** use navy, gray, or stock photography of server rooms and handshakes. Corporate IT vendor aesthetics signal mid-tier managed services, not special operations.
- **Don't** apply `border-radius` anywhere. Zero. Every rounded corner breaks the mechanical rigidity of the SCIF terminal.
- **Don't** use `box-shadow` for elevation. Prohibited. The tonal stack is the only elevation system.
- **Don't** use gradient text (`background-clip: text`). Solid color only. Weight and scale create emphasis, not gradients.
- **Don't** use `backdrop-filter: blur()` decoratively. Not part of this system.
- **Don't** put two crimson CTAs in visual competition. One primary CTA per section. The second option is always ghost.
- **Don't** write body copy in white (`#fafafa`). Body reads in secondary tone (`#a1a1aa`) so display headlines dominate by contrast.
- **Don't** use border-left greater than 1px as a colored accent stripe on cards or list items. The one permitted left border treatment is the testimonial quote mark (`border-l-2 border-crimson/60`), which pairs with an explicit semantic role (blockquote). Isolated side-stripes are a generic UI cliché.
- **Don't** use the hero-metric template (big number, small label, supporting stats, gradient accent). The HUD panels in this system use the same data format but they're earned by the WebGL substrate and HUD framing. Copy the data structure without the WebGL context and it reads as SaaS dashboard slop.
